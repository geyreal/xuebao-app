## 一、技术选型

### 1. 总体技术策略

项目需覆盖 Android、iOS、管理后台（Web），且为**单人开发**场景。技术选型的核心原则：

- **一套代码覆盖多端**，减少重复开发
- **技术栈成熟稳定**，社区资源丰富，降低踩坑风险
- **部署运维简单**，适合小团队低成本运行
- **依赖精简**，非必要不引入第三方 npm 库，优先使用 Vue 3 原生能力，降低维护成本和版本冲突风险

### 2. 各层技术选型

| 层次 | 技术方案 | 选型理由 |
|------|----------|----------|
| **移动端（Android + iOS）** | uni-app (Vue 3 + TypeScript) | 与管理后台共享 Vue 3 技术栈，单人开发零额外学习成本；编译为原生 APP，性能满足校内场景；国内教育类 APP 生态成熟，插件丰富 |
| **移动端状态管理** | Vue 3 原生 reactive 模块 | 使用 Vue 3 自带的 `reactive`/`computed` 导出全局状态模块，无需引入 Pinia 等第三方状态管理库，轻量且零额外依赖 |
| **管理后台（Web）** | Vue 3 + Element Plus | 轻量、上手快，Element Plus 提供完整后台 UI 组件（表格、表单、弹窗），适合单人快速搭建管理后台 |
| **后端服务** | Go (Gin 框架) | 编译为单二进制部署，高性能低资源占用；Gin 轻量且社区广泛；静态类型保障代码质量 |
| **数据库** | MySQL 8.0 | 成熟稳定的关系型数据库，班级隔离本质是关系查询过滤，SQL 天然适合；免费开源 |
| **缓存** | Redis | 用于 Token 存储、验证码、热点数据缓存（如公告列表），减轻数据库压力 |
| **文件存储** | 阿里云 OSS / 腾讯云 COS | 低成本对象存储，按量付费；CDN 加速；SDK 成熟，支持分片上传、直传 |
| **消息推送** | 极光推送 (JPush) | 免费额度充足（校内场景足够），同时覆盖 Android/iOS，接入简单 |

### 3. 依赖精简原则

移动端（uni-app）项目仅依赖以下 npm 包：

| 分类 | 依赖 | 说明 |
|------|------|------|
| **框架核心** | `@dcloudio/uni-app` 系列, `vue` | uni-app 跨平台框架本身，不可替代 |
| **构建工具** | `vite`, `typescript`, `@dcloudio/vite-plugin-uni` | 开发编译工具链，不可替代 |

以下功能**不引入第三方库**，使用原生实现：
- **状态管理**：Vue 3 `reactive` + `computed` + 模块导出（替代 Pinia）
- **国际化**：校内场景仅需中文，无需 vue-i18n
- **UI 组件**：使用 uni-app 内置组件 + 自定义组件，不引入第三方 UI 库

### 4. 开发工具链

| 工具 | 用途 |
|------|------|
| Git + GitHub/Gitee | 代码版本管理 |
| HBuilderX | uni-app 开发 IDE，支持一键编译到 Android/iOS |
| Docker + Docker Compose | 后端本地开发与生产环境一致性 |
| Nginx | 反向代理、静态资源托管、HTTPS |

### 2. 核心架构决策

#### 单体服务架构 vs 微服务

**选择单体架构**。理由：
- 单人开发，微服务增加部署和调试复杂度，得不偿失
- 用户规模 500-2000 人，单体完全承载
- 通过 Go 的 package 划分实现模块解耦，后期若需拆分可平滑迁移

#### 前后端分离

**选择前后端分离**。后端提供纯 RESTful API，前端（uni-app / Vue 管理后台）独立开发部署，职责清晰。全前端统一 Vue 3 技术栈，可共享工具函数和类型定义。

## 五、权限模型设计（核心）

班级隔离是本系统的**核心架构约束**。权限控制分为三层，在中间件中统一拦截。

### 1. 三层权限模型

```
┌─────────────────────────────────────────────────────┐
│ 第 1 层：身份认证 (Authentication)                    │
│ · JWT Token 校验，未登录直接拒绝                      │
│ · Token 中携带：user_id, role, class_ids             │
├─────────────────────────────────────────────────────┤
│ 第 2 层：角色鉴权 (RBAC)                              │
│ · 按角色控制 API 访问权限                             │
│ · 学生：只可调用查看类接口                            │
│ · 教师：可调用资源管理、打卡管理接口                   │
│ · 管理员：可调用所有接口                              │
├─────────────────────────────────────────────────────┤
│ 第 3 层：班级隔离过滤 (Class Isolation)                │
│ · 所有涉及 class_id 的查询，自动注入当前用户的班级范围 │
│ · 学生：WHERE class_id = 用户所在班级 ID              │
│ · 教师：WHERE class_id IN (用户负责的班级 IDs)        │
│ · 管理员：不过滤（全局可见）                          │
└─────────────────────────────────────────────────────┘
```

### 2. 中间件实现要点

```go
// 伪代码：班级隔离中间件
func ClassIsolation() gin.HandlerFunc {
    return func(c *gin.Context) {
        user := GetCurrentUser(c)
        
        switch user.Role {
        case RoleStudent:
            // 学生只能访问自己所在班级
            c.Set("allowed_class_ids", []uint64{user.ClassID})
        case RoleTeacher:
            // 教师可访问所负责的所有班级
            c.Set("allowed_class_ids", user.ClassIDs)
        case RoleAdmin:
            // 管理员不限制
            c.Set("allowed_class_ids", nil) // nil 表示不过滤
        }
        c.Next()
    }
}
```

### 3. JWT Token 载荷设计

```json
{
  "user_id": 10001,
  "account": "2024001",
  "role": 1,
  "class_ids": [5],
  "exp": 1711900800,
  "iat": 1711814400
}
```

> Token 有效期 7 天，续签策略：距过期不足 1 天时自动续签。

## 七、文件存储方案

### 1. 上传流程（客户端直传 OSS）

```
┌──────────┐   1. 请求签名URL    ┌──────────┐   2. 生成签名    ┌──────────┐
│ uni-app   │ ───────────────► │  Go API  │ ──────────────► │   OSS    │
│  客户端    │                   │  Server  │                 │  服务    │
│           │ ◄─────────────── │          │                 │          │
│           │   返回签名URL      │          │                 │          │
│           │                   └──────────┘                 │          │
│           │   3. 直传文件                                   │          │
│           │ ──────────────────────────────────────────────► │          │
│           │   4. 上传成功                                   │          │
│           │ ◄────────────────────────────────────────────── │          │
│           │                   ┌──────────┐                 │          │
│           │ 5. 提交资源元数据  │  Go API  │                 │          │
│           │ ───────────────► │  Server  │                 │          │
└──────────┘                   └──────────┘                 └──────────┘
```

> **设计要点**：文件不经过后端服务器，直传 OSS，避免后端成为瓶颈。后端仅负责生成临时签名和保存资源元数据。

```

│   ├── image/      {class_id}/{uuid}.jpg
│   └── audio/      {class_id}/{uuid}.mp3
└── avatars/        (预留)
```

## 九、关键设计决策总结

| # | 决策项 | 决策 | 理由 |
|---|--------|------|------|
| 1 | 架构风格 | 单体服务 | 单人开发、用户规模小，微服务增加复杂度无收益 |
| 2 | 移动端方案 | uni-app (Vue 3) 跨平台 | 与管理后台统一 Vue 3 技术栈，单人开发零额外学习成本，一套代码覆盖 Android/iOS |
| 3 | 班级隔离实现 | 中间件 + SQL 过滤 | 统一拦截，所有查询自动注入 class_id 过滤，避免遗漏 |
| 4 | 文件上传 | 客户端直传 OSS | 不经过后端，避免带宽瓶颈，支持大文件 |
| 5 | 认证方案 | JWT 无状态 | 适合移动端，服务端无需维护 Session |
| 6 | 通知推送 | 极光推送 | 免费额度足够校内场景，接入简单 |
| 7 | 部署方式 | Docker Compose 单机 | 运维简单，一键启停，适合单人维护 |
| 8 | 数据库 | MySQL 单实例 | 数据量小（万级），无需分库分表 |
